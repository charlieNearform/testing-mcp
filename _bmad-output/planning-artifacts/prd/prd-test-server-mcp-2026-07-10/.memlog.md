- (change) Added C14-C17 (async run_tests, live per-test progress, log tail, stall watchdog) during interactive design session 2026-07-17, driven by real production pain: an 800+s run held one MCP tool call open the whole time (fragile against client-side request timeouts / idle-stream timeouts), had zero visibility into what was actually happening, and had no protection against a genuinely wedged worker process.
- (capability) Capability C14: Async-capable run_tests — returns the full result synchronously if the run finishes within a configurable grace period (waitMs: per-call arg -> project config -> daemon config -> 10s built-in default; explicit null at any layer = wait forever, a deliberate opt-out); otherwise returns {runId, projectId, state:"running"} and the caller polls the already-existing get_test_status tool, extending the same kick-off-then-poll pattern already documented for start_watch. Success: a run exceeding the grace period never holds an MCP tool call open past that window; a fast run's behavior is unchanged from today.
- (capability) Capability C15: Live per-test progress — while a run is in flight, get_test_status and the human monitoring UI expose a live, bounded list of tests with status pending/running/passed/failed/skipped, populated via Vitest's onTestCaseReady/onTestCaseResult reporter hooks (Vitest 3+; gracefully degrades to today's file-level-only granularity when a project's Vitest predates these hooks, since Vitest itself simply never calls them). Success: a human or agent watching a long run can see which specific test is currently executing and which have already passed/failed, not just a file count.
- (capability) Capability C16: Console log tail with follow — the daemon captures the worker's stdout+stderr (previously discarded/inherited) into a bounded per-run ring buffer (last 1000 lines), exposed via the UI with a follow mode, while still forwarding stderr through to the daemon's own stderr unchanged. Success: if a suite finishes all tests but the process doesn't exit, this is directly observable (all tests show terminal status, log tail goes idle, run state stays "running") instead of being an invisible hang.
- (capability) Capability C17: Stall watchdog, on by default — the worker is killed automatically if there has been no test-level progress signal for testTimeout + grace (test timeout read from the project's own resolved Vitest config; grace configurable, default 5s). Unlike the existing opt-in runTimeoutMs (a whole-run cap, unbounded by default since a suite can legitimately run 15-20+ minutes), this targets a genuine hang/zombie-process bug and is on by default; requires a coverage-measurement-phase heartbeat as a companion mechanism since that phase emits zero progress signals today and would otherwise false-positive. Success: a genuinely wedged worker is killed with a clearly-distinguishable error message within testTimeout+grace of its last real progress, never earlier, never silently hanging forever.
- (constraint) Stall watchdog (C17) must reuse the orchestrator's existing worker-kill mechanism (the same finish() helper already used by the opt-in runTimeoutMs timeout) — no second kill path. All new worker<->daemon IPC message types (C14-C17) are additive to the existing ToWorker/FromWorker discriminated union; no breaking change to existing message shapes.
- (constraint) Per-run live state introduced by C15/C16 (live test list, log ring buffer) is in-memory only and transient, cleared when a run settles — consistent with the existing runState in-memory pattern, not persisted to <git-root>/.test-mcp/ history (which remains the durable end-of-run record). /ui* routes stay strictly read-only, loopback + Host/Origin gated, unauthenticated per the existing documented invariant — none of C14-C17 introduces a new mutation surface.
- (note) Non-goal (C14-C17): does not address the MCP SDK client's own default request timeout (DEFAULT_REQUEST_TIMEOUT_MSEC=60000) — a client-side concern (e.g. an MCP client's own internal request-timeout handling), not something this daemon controls. C14's grace-period design makes this moot for the default/recommended configuration; an operator who explicitly configures "always wait" is knowingly opting back into that client-side risk.
