updated: 2026-07-16T12:22

- (event) Brief originates from a design discussion validating whether test-mcp's incremental-testing/coverage story holds up; surfaced that Vitest is hardcoded (vitest/node API in src/worker/index.ts) and selection granularity is file-level only. User asked to make runner support pluggable.
- (decision) Vitest becomes the first default RunnerPlugin implementation, extracted with no behavior change - not a rewrite.
- (decision) Coverage is an optional, graded per-plugin capability (none/summary/line-hit), not assumed - some runners (e.g. Docker/visual) won't support it at all.
- (decision) Design precedent: the ImpactProvider seam proposed in the Story 6.9 CRG spike (optional, probed, unions in, never narrows, fails closed) is the model for the RunnerPlugin seam.
- (assumption) [ASSUMPTION] A generic shell-command/subprocess plugin (incl. Docker) is the escape hatch for non-JS runners, rather than bespoke first-party plugins for every ecosystem.
- (event) Known cost flagged: container/subprocess runners add cold-start latency in tension with the fast-incremental value proposition - likely an opt-in slower tier.
- (event) Research digest landed: Jest's testRunner module-swap is the closest existing analogue to RunnerPlugin (1:1 engine swap, not pytest-style N:1 hook fan-out); CTRF is the best-fitting precedent for a JSON plugin-contract shape; lcov confirmed genuinely polyglot; Bazel/Nx are the closest polyglot-orchestration comparables but neither treats coverage as optional/graded per-plugin - that framing is the real gap. Full detail captured in addendum.md.
- (decision) Sequencing: make plugins POSSIBLE first (seam + Vitest as the sole reference implementation, extracted with no behavior change). Jest is the planned second plugin, but its purpose is to validate the abstraction holds under a second real runner - not to reach feature parity with Vitest. No other runners in this brief's scope.
- (decision) Scope expansion (user-raised, core not incidental): a registered project may run multiple distinct test suites (e.g. Vitest for unit, Playwright for e2e), each potentially needing a different runner plugin. 'One plugin per project' is insufficient - the registration/config model must support multiple named suites per project, each bound to its own plugin.
- (assumption) [ASSUMPTION] test-mcp register detects which plugins' config files/markers are present in a project and interactively prompts the user to confirm/select which suite(s) to register with which plugin, rather than requiring fully manual suite declaration.
- (event) Drafted brief.md v1 (fast path) with 4 [ASSUMPTION] tags: coverage-merge-format deferred, interactive suite detection in register, generic shell/Docker plugin out of scope this increment, function-level selection tracked separately (not this brief). Awaiting user review of assumptions before finalize.
- (decision) Assumption 1 CONFIRMED: cross-runner coverage merge format is deferred to a later increment; this increment's bar is honest per-plugin capability reporting only.
- (decision) Assumption 2 REFINED (not just confirmed): test-mcp register auto-detects suite/plugin config markers as the primary path, but supports an optional explicit suite definition a user can supply when auto-detect fails or picks wrong - auto-detect is not the only path.
- (decision) Assumption 3 CONFIRMED: generic shell-command/Docker plugin explicitly deferred to a later discovery task; this brief focuses on JS-ecosystem runners (Vitest, Jest) only.
- (decision) Assumption 4 CONFIRMED, with rationale: function-level selection stays tracked separately from this brief (toward parked CRG/Story 6.9 work), and per the user may NEVER be added as few runner plugins would likely be able to support that granularity anyway - not just a sequencing choice but a plausible permanent scope boundary.
- (event) Finalized: structure + prose editorial passes applied to brief.md and addendum.md. Brief status remains draft pending user's final read-through; no external handoffs configured for this workflow.
