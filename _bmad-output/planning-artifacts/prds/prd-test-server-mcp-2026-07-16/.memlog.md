- (event) Canonical PRD lives at docs/prd.md (not duplicated into this workspace) per repo convention (CLAUDE.md: docs/ paths are authoritative, not to be relocated). This workspace holds only the update run's audit trail.
- (decision) Phase 1 scope is JavaScript/TypeScript projects only, with Vitest as the sole supported test runner, integrated via the vitest/node programmatic API (startVitest/createVitest) and never by parsing CLI output.
- (decision) Jest and pytest support are explicitly deferred to Phase 3; they are out of scope for Phase 1/2 and will not be built until the core coverage-based selection engine is proven on Vitest.
- (decision) Coverage-based test selection is deliberately positioned as table-stakes, not the product's differentiator (multiple competitors already do it for Vitest/Jest); the differentiator is the delivery architecture: one always-on multi-project daemon, per-project Vitest version isolation, transparent repo-local state, and setup-baseline subtraction.
- (decision) Decision (2026-07-10): rather than wrapping the testpick CLI (which owns the test run) or reinventing the algorithm from scratch, port/vendor testpick's MIT-licensed V8 snapshot-diff coverage-attribution algorithm into our own per-project worker, retaining its MIT copyright/license notice, and add setup-baseline subtraction ourselves.
- (decision) Product is structured as two layers: Layer 1 (MCP server, programmatic-only, AI-agent-facing) is the primary focus and ships in Phase 1; Layer 2 (human web UI) is explicitly secondary/pure-convenience, cannot function standalone, and is deferred to Phase 2 (Epic 5).
- (decision) Phase 1 MVP scope is fixed to Epics 1-4: singleton daemon lifecycle, MCP server + project registration, Vitest integration (with watch mode and isolation verification), coverage intelligence (git-delta + coverage-map selection), and output/status/dry-run — this phasing is the single source of truth over the product brief and brainstorm docs.
- (decision) Phase 2 scope is priority scoring (git recency + failure history), test health monitoring, and the Human Monitoring UI (Epic 5); these are deliberately excluded from the Phase 1 MVP.
- (decision) Phase 3/deferred scope covers Jest & pytest support, priority-based sharding, IDE integration, trend analytics, suggested fixes, and dependency-graph visualization, plus fixture/setup-time cost tracking, ordering-dependency detection, and parallel resource-contention quotas — the latter three were raised in brainstorming as 'must-haves' but were deliberately deferred as research-grade until the core selection engine is proven.
- (decision) Architecture commits to a single always-on singleton daemon (enforced via lockfile + known port in a central dir) serving multiple registered projects over Streamable HTTP, chosen deliberately over per-call stdio wrappers or per-project config plugins used by competing tools.
- (decision) Tests execute in a per-project worker subprocess (CWD = project root) that resolves vitest/node from the project's own node_modules, never the daemon's, so the daemon/CLI version is fully decoupled from any registered project's installed Vitest version and mixed-version monorepos don't break.
- (decision) State model deliberately splits per-project state (coverage map, run history, config.json) git-ignored under <git-root>/.test-mcp/ (visible/diffable, not a hidden DB) from daemon-global state (registry, pid, port, lockfile) under a central dir like ~/.test-mcp/ — the two must never be written into each other's location.
- (decision) Test isolation relies entirely on Vitest's built-in isolation (isolate: true, the Vitest default) rather than building a bespoke isolation engine; if a project explicitly disables isolation, the server surfaces that fact in run metadata rather than compensating for it.
- (decision) The daemon must pin its Vitest version dependency deliberately: the vitest/node advanced programmatic API signature differs between Vitest 3.x and 4.x, and runTestFiles specifically requires Vitest 4.1+.
- (decision) Selection philosophy is 'never silent skip, prioritise recall': git --changed (static import graph) and the coverage-map (runtime V8 snapshot-diff) selections are unioned; genuinely unbounded changes force a full-suite run, while bounded-but-uncertain changes run the tight set and report degraded confidence so the agent can choose to run a full pass, instead of always defaulting to full-suite on any uncertainty.
- (decision) Incremental change detection compares against the last run by default using a git-free content-hash approach (aligning with the proposed native , vitest#9917), with git diff/--changed against HEAD retained as an available alternative rather than the primary mechanism.
- (decision) Correction to prior entry (shell mangled it): Incremental change detection compares against the last run by default using a git-free content-hash approach, aligning with the proposed native vitest --stale feature (vitest#9917), with git diff/--changed against HEAD retained as an available alternative rather than the primary mechanism.
- (decision) Platform scope is macOS-first for Phase 1; Linux and Windows support is explicitly deferred to Phase 2, not a Phase 1 acceptance requirement.
- (decision) Success metrics are recorded as aspirational/directional targets, not committed SLAs, to be reconfirmed per project: dry-run plan generation <5s, incremental runs <15s for single-file changes, selection precision target 99%+, coverage-map buildable within one full run covering 95%+ of suites, and 80%+ team adoption/replacement of existing test-running setup within 6 months.
