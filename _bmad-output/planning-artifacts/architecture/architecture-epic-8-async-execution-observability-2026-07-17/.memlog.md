- (decision) AD-17 Async run_tests (fulfills C14): Orchestrator.startRun(project,opts) returns {runId, result:Promise<TestResult>} synchronously -- runId generated at call time, not inside executeWorker. run_tests MCP handler resolves waitMs (per-call arg -> project .test-mcp/config.json defaultRunWaitMs -> daemon config defaultRunWaitMs -> built-in 10s; explicit null at any layer = wait forever) and races the run against that window: returns the full TestResult unchanged if it settles first, otherwise {runId, projectId, state:'running', message} while the run keeps executing in the background. runTests()/runPlan() become thin wrappers over startRun so every existing caller/test is unaffected. Binds: run_tests MCP tool, Orchestrator public API. Prevents: a long run holding one MCP tool call open past a client's own request timeout; a caller unable to recover a runId for a still-running job. Rule: get_test_status is the ONLY poll surface (no new tool) -- extends the same pattern already documented for start_watch, per AD-1's 'programmatic interface first, one coherent surface' spirit.
- (decision) AD-18 Live per-test progress (fulfills C15): worker wires onTestCaseReady/onTestCaseResult (Vitest 3+ optional reporter hooks) alongside the existing onTestModuleEnd, sending case-start/case-result IPC messages; a project whose Vitest predates these hooks simply never triggers them (no explicit feature-detection needed on our side -- Vitest itself decides). Orchestrator keeps a bounded (2000-entry) per-project live test list, keyed (file,name), cleared when the run settles. Binds: worker reporter wiring, orchestrator live state. Prevents: false live-view claims when Vitest can't support them; unbounded memory growth from a huge suite's test list.
- (decision) AD-19 Bounded live log capture (fulfills C16): fork()'s stdio moves from [ignore,ignore,inherit,ipc] to [ignore,pipe,pipe,ipc]. stderr is teed -- captured into a bounded (1000-line, 4000-char/line) per-project ring buffer AND still written through to the daemon's own process.stderr unchanged (no regression for existing daemon-log-based debugging). stdout is captured into the SAME ring but NOT also forwarded anywhere else (new visibility, not new noise). Binds: orchestrator executeWorker, live state. Prevents: an operator's existing daemon-stderr-based debugging silently breaking; unbounded memory growth from a chatty suite.
- (decision) AD-20 Stall watchdog, on by default (fulfills C17): worker reads the project's resolved Vitest testTimeout once via a lightweight createVitest() discovery call (same pattern as the existing readCoverageThresholds) before the real run starts, reports it via a new config IPC message. Orchestrator arms a timer at testTimeout+staleTestGraceMs (default grace 5000ms), reset on every test-level progress signal (case-start/case-result, or module-level progress when case hooks are unsupported, or the coverage-phase heartbeat below); on fire, kills the worker via the SAME finish() helper the existing opt-in whole-run runTimeoutMs cap already uses -- no second kill path -- producing a WorkerError message distinguishable from both the runTimeoutMs-cap message and the exit(code,signal) message. The coverage-measurement phase (buildAndPersistCoverageMap's silent per-file measurement reporter) currently emits zero progress signals for its entire duration; a new phase-progress IPC message (coverage phase only) is a REQUIRED companion, not optional -- without it, any coverage-enabled run on more than a couple of files is a false-positive stall-kill waiting to happen. Binds: worker, orchestrator kill path. Prevents: a genuinely wedged worker (e.g. vitest.close() never resolving) hanging forever with no operator visibility; a false-positive kill during the silent coverage phase; a second, divergent kill mechanism alongside the existing timeout path.
- (decision) AD-21 MCP/UI live-state parity (reconciles AD-1 'UI cannot expose anything MCP doesn't'): get_test_status's MCP response returns the FULL bounded live state (up to the same 2000-test/1000-log-line caps as the in-memory ring, no further truncation) on every on-demand call -- there is no MCP-side reason to slice it smaller, since a poll is a deliberate single-project request, not a broadcast. The human UI's /ui/events SSE snapshot -- a repeated push to every connected browser tab across every running project on every single test event -- slices to a much smaller live view (last 20 log lines, most-recent 200 tests) purely for push-payload/frequency reasons; the UI's dedicated /ui/api/projects/:id/log (one-shot) and /log/events (follow) routes surface the SAME full data the SSE snapshot elides, so nothing is UI-exclusive -- an MCP caller gets full parity by polling get_test_status repeatedly instead. Binds: get_test_status payload, ui/index.ts snapshot + new log routes. Prevents: a live-state fidelity gap where a human watching the UI sees something an agent polling MCP structurally cannot ever see.
- (constraint) All new worker<->daemon IPC message types (config, case-start, case-result, phase-progress) are additive to the existing ToWorker/FromWorker discriminated union in src/types/ipc.ts, each with its own Zod schema variant -- no breaking change to existing shapes, matching this repo's schemaVersion/additive-envelope discipline.
- (constraint) Per-run live state (test list, log ring) is in-memory only, transient, cleared when a run settles -- never persisted to <git-root>/.test-mcp/ history, which stays the durable end-of-run record per inherited AD-8. This is deliberately NOT a fourth persistence tier; it's ephemeral request-serving state, same category as the existing in-memory runState map.
- (constraint) /ui* routes (including the two new log routes) stay strictly read-only, loopback + Host/Origin gated, unauthenticated -- matching the existing documented invariant; none of AD-17 through AD-21 introduces a new mutation surface anywhere in the human UI.
- (note) Non-goal carried from SPEC.md: the MCP SDK client's own default request timeout (60s) is a client-side concern this daemon does not control; AD-17's grace-period design makes it moot for the default/recommended configuration, but an operator explicitly configuring waitMs=null (wait forever) knowingly opts back into that client-side risk.
- (direction) Deferred: a debounce on /ui/events push frequency (AD-21's SSE snapshot fires on every single test event across every running project today) -- not built this epic, flagged as a likely future follow-up if push frequency becomes a practical problem in production. Also deferred: any change to the MCP SDK client's own request-timeout behavior (out of this daemon's control, see non-goal above).
